# Sova AI｜指标异动归因工作台

Sova AI 是一个面向中文业务分析 / 数据分析场景的 AI 指标异动归因工作台。

“Sova” 有猫头鹰与侦察的意象，代表从复杂业务数据中观察异常信号、追踪隐藏线索，并形成可验证的证据链。

它不是普通的 Chat with CSV 工具，也不是简单图表生成器。它的核心目标是：当用户遇到一个模糊的业务指标异动问题时，系统先通过多轮业务澄清，把问题转化为可执行的数据分析任务；随后引导用户上传数据，使用 Pandas + DuckDB 执行分析，并生成结果表、图表、证据链和报告草稿。

## 当前指标口径生成逻辑

- 前端已新增 rule-based / mock Metric Definition Generator。
- 系统会根据用户输入的业务问题识别指标类型，并动态生成前三张候选指标口径卡片。
- 第四张“自定义口径”卡片固定保留。
- 优惠券核销率只是一个内置识别模板。
- 当前支持的 mock 指标类型：
  - 优惠券 / 核销率
  - DAU / 活跃
  - 转化率
  - GMV / 销售额
  - 留存
  - 退款
  - 无法识别时使用通用业务指标口径
- Stage 6B 已新增真实 LLM 指标识别与口径卡片生成：如果用户已在 API 设置中配置可用模型，点击“开始澄清”时会优先调用 `/api/llm/metric-definitions` 生成指标名称、指标类型、业务场景和前三张候选口径卡片。
- 如果未配置 API，或 LLM 调用失败 / 返回格式不合规，系统会自动回退到本地 rule-based 生成逻辑。
- 第四张“自定义口径”卡片始终固定保留，不由 LLM 生成。

## Stage 7A 已完成内容

- 新增 AI 动态业务澄清系统：`POST /api/llm/business-clarification`。
- 用户点击“开始澄清”时，如果已配置 API，会优先由 LLM 根据业务问题动态生成：
  - 指标口径卡片
  - 分析维度卡片
  - 近期变化因素卡片
  - 下一步数据需求
- 第四张“自定义口径”、分析维度中的“自定义维度”、变化因素中的“暂无明显变化 / 不确定”仍由系统固定保留。
- 未配置 API 或 LLM 失败时，会回退到本地规则版业务澄清生成器。
- 本地 fallback 已补充游戏表现、内容运营、教育学习、通用业务和电商退款等基础场景，避免非电商场景默认出现优惠券、GMV、商品、商家等无关词。
- 上传数据时会附带当前业务澄清上下文，后端 Schema 识别会结合业务场景判断“当前数据可支持的分析”和“当前数据暂不支持的部分”。
- 本阶段不让 LLM 生成 SQL，不改变 DuckDB 执行逻辑，也不删除已有 rule-based fallback。

## Stage 2A 对比周期确认

- 用户确认指标口径后，主交互区会进入“对比周期确认”步骤。
- 对比周期选项使用交互式卡片，包含“本周 vs 上周”“本月 vs 上月”“活动期 vs 活动前”“自定义对比周期”。
- 选择“自定义对比周期”后，需要填写具体周期才算确认。
- 确认对比周期后，右侧“分析就绪面板”会推进到“分析维度”阶段，并将“对比周期”加入已确认信息。

## Stage 2B 分析维度多选

- 用户确认对比周期后，主交互区会进入“分析维度选择”步骤。
- 分析维度选项继续使用交互式卡片，支持多选、取消选择和自定义维度。
- Stage 7A 后，分析维度卡片会优先由 AI 根据当前业务问题动态生成；未配置 API 或生成失败时使用本地规则 fallback，不再把固定电商维度默认套用到所有场景。
- 至少选择一个维度后，“确认分析维度”按钮才可用。
- 确认分析维度后，右侧“分析就绪面板”会保留在“分析维度”阶段，但状态会进入“近期变化因素确认”。

## Stage 2C 近期变化因素与数据准备

- 用户确认分析维度后，主交互区会进入“近期变化因素确认”步骤。
- 近期变化因素继续使用交互式卡片，支持多选、取消选择，并对“暂无明显变化”“不确定”做互斥处理。
- 确认变化因素后，右侧“分析就绪面板”会进入“数据准备”阶段。
- 数据准备阶段会展示“当前理解摘要”和“下一步需要的数据”提示。
- Stage 3A 已将原先的“下一阶段：上传数据”mock 按钮替换为真实上传入口。

## 前端运行方式

进入前端目录：

```bash
cd metricflow-ai/frontend
```

安装依赖：

```bash
npm install
```

启动开发服务：

```bash
npm run dev
```

默认访问：

```text
http://localhost:3000
```

如需让前端调用其他后端地址，可以设置：

```bash
NEXT_PUBLIC_API_BASE_URL=http://localhost:8000
```

## 后端运行方式

进入后端目录：

```bash
cd metricflow-ai/backend
```

创建并启用虚拟环境后安装依赖：

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

启动 FastAPI：

```bash
uvicorn main:app --reload --port 8000
```

健康检查：

```text
http://localhost:8000/health
```

## Stage 3A 已完成内容

- 数据准备阶段已经提供真实“上传数据”入口，不再只是“下一阶段：上传数据”的 mock 按钮。
- 前端支持点击或拖拽上传 CSV、Excel（.xlsx / .xls）文件，当前支持单文件和多文件的表单结构。
- 后端新增 `POST /api/data/upload`，使用 Pandas 读取 CSV / Excel，并返回上传批次 `upload_id`。
- 后端会保存上传文件到 `backend/uploads/{upload_id}/`，该目录仅用于后续本地分析阶段衔接，不是数据库持久化。
- 后端会清洗字段名，保留原始字段名和清洗后字段名，并返回字段类型、缺失率、样例值和前 5 行样例数据。
- 前端会用卡片和表格展示“数据识别结果”，包括文件名、表名、行数、字段数量、字段列表、字段类型、缺失率和样例行。
- 系统会基于字段名规则判断“当前数据可支持的分析”。Stage 7A 后，该判断会结合当前业务澄清上下文，避免在非电商场景默认提示金额、优惠券、商品、商家等无关分析。
- 系统会结合用户已选择的分析维度提示“当前数据暂不支持的部分”，只提示当前场景相关的缺失项。
- 上传成功后右侧“分析就绪面板”会补充“数据状态：已上传并完成字段识别”。
- “生成分析计划”按钮已经在 Stage 3B 升级为规则版真实功能，会基于业务澄清结果和上传 schema 生成结构化分析计划。

## Stage 3B 已完成内容

- 后端新增 `POST /api/analysis/plan`，基于业务问题、指标口径、对比周期、分析维度、近期变化因素和上传 schema 生成规则版分析计划。
- 分析计划会进行字段匹配，识别时间字段、指标相关字段、用户字段、订单字段、城市 / 地区字段、渠道字段、金额字段、优惠券字段、用户类型字段、商家 / 商品字段等。
- 分析计划会根据用户选择的分析维度生成分析步骤，例如按用户类型、地区 / 城市、渠道来源、商品 / 商家 / 内容或时间粒度拆解指标变化。
- 每个分析步骤会返回 `ready`、`partial` 或 `blocked` 状态，前端展示为“可执行”“部分支持”“暂不支持”。
- 前端“生成分析计划”按钮现在会调用后端接口，并在上传结果之后展示“分析计划”区域。
- “分析计划”区域包含分析目标、指标与周期、字段匹配情况、分析步骤、当前限制和下一步说明。
- 如果尚未上传数据，点击“生成分析计划”会提示“请先上传数据并完成字段识别。”
- Stage 3B 当时仍然不执行 DuckDB SQL；Stage 4A 已补充固定规则 DuckDB 基础执行。

- 前端点击“生成证据链”时会优先检查浏览器中的 API 设置；未配置 API 时继续调用 `/api/analysis/evidence` 使用本地规则，并给出中文提示。
- LLM 只总结现有 `execution_result` 中的结果表，不重新计算数据，不生成 SQL，不生成报告，也不输出最终因果结论。
- 生成证据链后，现有报告草稿流程保持不变；Stage 6E 已支持在配置 API 时优先生成 LLM 报告草稿。

## Stage 6E 已完成内容

- 前端点击“生成报告草稿”时会优先检查浏览器中的 API 设置；未配置 API 时继续调用 `/api/analysis/report` 使用本地规则，并给出中文提示。
- LLM 只整理现有业务信息、结果表、证据链和限制，不重新计算数据，不生成 SQL，不输出最终因果结论。

## LLM 模型兼容说明

- GPT-5 系列模型使用兼容参数：请求中不发送 `temperature` / `top_p` / `max_tokens`，改用 `max_completion_tokens`，并默认传 `reasoning_effort: "none"`。
- gpt-4.1、gpt-4o、DeepSeek 和其他 OpenAI-Compatible 模型继续使用原有 `max_tokens` + `temperature` 请求方式。
- 推荐模型：`gpt-4.1`、`gpt-4o`、`gpt-5.2`、`gpt-5-mini`。
- 如果模型返回 HTTP 400，请优先检查模型 ID 是否真实可用，以及请求参数是否与该模型兼容。
